# Report — 31.03.2026

## Summary

- ✨ Status filter moved to the shared filters bar (visible in both Tasktracker
  and Kanban tabs)
- 🔧 Issues list reloads first page client-side when shared filters change (no
  full page navigation)
- 🐛 Fixed misleading `getMethodologies failed` error message in participants
  API
- 🐛 Follow-ups analysis page no longer crashes on 404 from
  participants/profiles endpoints
- 🏗️ `lint` / `lint:fix` scripts switched from `next lint` to bare `eslint .`

---

### ✨ New features

- **Status filter in shared filters bar**: The status dropdown was previously
  rendered inside the Issues (Tasktracker) tab only and was not part of the
  shared filter set. It has been moved into `SharedFiltersBar`, so the same
  status filter is now available (and synced to the URL) for both the
  Tasktracker and Kanban tabs. `SharedFilters` type now includes a `status`
  field.

### 🔧 Improvements

- **Client-side first-page reload in Issues list**: When shared filters or sort
  order change, the Issues list now fetches and replaces the first page of
  results in place instead of relying solely on SSR re-navigation. This removes
  the visible flicker that occurred when the page count changed after filtering.
  Sort and filter state are kept in a `useRef` so the `fetchMore` callback does
  not need to close over reactive values.

- **URL sync moved outside setState callback**: `handleFiltersChange` in
  `TasktrackerTabs` previously called `updateUrl` inside a `setState` updater,
  which is a React anti-pattern (side effects in render). URL updates now happen
  in a dedicated `useEffect` that runs after state settles, and `status` is
  included in the synced URL params.

### 🐛 Bug fixes

- **Misleading error name in participants API**: `getAttendees` and `getGuests`
  both threw `"getMethodologies failed: ..."` on error — a copy-paste bug from
  an unrelated file. The error prefixes are now corrected to
  `"getAttendees failed"` and `"getGuests failed"` respectively, making log
  triage accurate.

- **Follow-ups analysis page crash on 404**: `EventOverview` used `Promise.all`
  to fetch attendees and guest profiles. If either endpoint returned 404 (e.g.
  the CalendarEvent is not accessible to the current user via the participants
  route), the whole component threw and crashed the page. The call has been
  changed to `Promise.allSettled` so a 404 on participants or profiles results
  in empty arrays rather than an unhandled rejection. Two regression tests cover
  this path.

### 🏗️ Infrastructure

- **ESLint scripts**: `lint` and `lint:fix` in `package.json` now invoke
  `eslint .` directly instead of `next lint`, giving consistent behaviour
  outside the Next.js CLI and aligning with the flat-config
  (`eslint.config.mjs`) setup already in place.
